PILAR I: Diseño de Casos de Prueba

1. Prioridad: Comprensibilidad

  • Un programa fácil de entender es más fácil de codificar y depurar.
  • Si no es comprensible, es difícil corregir.

2. Pruebas Empíricas Metódicas

  • Benchmark: Conjunto de datos de referencia para medir las optimizaciones.
  • No confiar en la primera prueba.

3. Análisis de Rendimiento (Casos de Entrada)

  • La eficiencia depende de la entrada.
  • Analizar: Mejor Caso, Peor Caso y Caso Promedio.
  • Diseñar pruebas que reflejen la distribución real de los datos.
1. Comprensibilidad (5 min): El código limpio es el primer paso. 2. Empíricas (5 min): No es "lo que funcione", es "lo que demuestro que funciona" con benchmarks. 3. Casos (5 min): La eficiencia del algoritmo varía. Debemos probar los límites (Peor Caso) que romperían nuestro proyecto.

PILAR II: Depuración y Robustez

Buscamos abordar los Race Conditions (condiciones de carrera) que hacen fallar algoritmos.

1. Programación Sólida

  • Comprobación de Parámetros: Verificar al inicio de la función (ej. ¿el contenedor está vacío antes de eliminar?).
  • Assert: Usarlos para clarificar cómo debe operar el código internamente, pero no como manejo de errores para el usuario final.
  • Evitar `null`: Usar operadores de cortocircuito (`&&` o `||`) para evitar evaluar atributos de objetos que podrían ser `null`.

2. Manejo de Excepciones (`try-catch`)

  • En programas reales, es la mejor práctica para tratar errores de datos de entrada o fallos externos.
  • Es vital detectar overflows y underflows en estructuras (ej. colas).
Depuración y Excepciones (20 min): No solo corregir, sino hacer el código defensivo. Mostrar ejemplo rápido de try-catch en el lenguaje del proyecto. Enfatizar la diferencia entre un Assert (que mata el programa si se viola) y un try-catch (que recupera el flujo).

Ejemplo: Uso de `assert()`

Utilizado para comprobar precondiciones internas. Si la condición falla, el programa termina.


#include <assert.h>
#include <stdio.h>

// Función que calcula el factorial
int factorial(int n) {
    // n debe ser positivo o cero. Si no, hay un error de lógica.
    assert(n >= 0 && "El factorial no puede ser 
    calculado para números negativos."); 
    
    if (n == 0) {
        return 1;
    }
    return n * factorial(n - 1);
}

int main() {
    printf("Factorial de 5: %d\n", factorial(5));
    // El siguiente caso fallaría si se ejecuta en modo DEBUG:
    // factorial(-1); 
    return 0;
}

Ejemplo Manejo de nulls

Evitar los "Segmentation Faults" comprobando si la asignación de memoria o la entrada de puntero ha fallado.


#include <stdio.h>
#include <stdlib.h>

int inicializar_estructura(void *ptr) {
    // COMPROBACIÓN: Si el puntero de entrada es NULL, devuelve un error.
    if (ptr == NULL) {
        fprintf(stderr, "Error: Puntero de entrada NULL.\n");
        return -1; // Usar código de error
    }
    // ... Lógica de inicialización ...
    return 0; // Éxito
}

int main() {
    int *data = (int *)malloc(sizeof(int) * 10);
    
    // COMPROBACIÓN: malloc puede fallar si no hay memoria disponible.
    if (data == NULL) {
        fprintf(stderr, "Error fatal: Fallo al asignar memoria.\n");
        // En un programa real, aquí se liberan recursos y se sale.
        return EXIT_FAILURE; 
    }

    inicializar_estructura(data);
    
    free(data);
    return EXIT_SUCCESS;
}

Ejemplo Manejo de Errores con Códigos de Retorno

El estándar de la programación en C para manejar errores de E/S o de función (simula la lógica de `try-catch`).


#include <stdio.h>

// Definición de códigos de error
#define ERR_DIV_CERO -1
#define EXITO 0

int dividir(int a, int b) {
    if (b == 0) {
        return ERR_DIV_CERO; // Retorna el código de error en lugar del resultado
    }
    return a / b;
}

int main() {
    int resultado = dividir(10, 0);

    // Simulación de 'catch': Comprobar el código devuelto
    if (resultado == ERR_DIV_CERO) {
        printf("¡ERROR! (Catch): Se intentó una división por cero.\n");
    } else {
        printf("Resultado (Try): %d\n", resultado);
    }
    
    return 0;
}

PILAR III: Optimización y Código Limpio

Máxima: "Primero ajusta el Algoritmo, luego ajusta el Código."

A. Prioridad: Algoritmos y Estructuras (El Gran Impacto)

  • La eficiencia se basa en buenos algoritmos y organización de la información (Estructuras de Datos).
  • Compensación (Trade-off): A menudo se reduce el tiempo aumentando el espacio
  • Eliminar Recursión: Las llamadas recursivas son costosas. Sustituir por iteración o una pila explícita puede ser más rápido.

B. Ajuste Fino del Código (Code Tuning)

  • Evitar Trabajo, No Acelerar: Es más eficiente no hacer un cálculo costoso que hacerlo rápido (ej. usar `bounding box` en gráficos 3D).
  • Recolección de Estadísticas: Usar Profilers para saber dónde invertir el esfuerzo.
  • Legibilidad: La mayoría de los "ajustes finos" son simplemente limpiar un código descuidado, no aplicar trucos ilegibles.
  • Confía en el Compilador: Los compiladores modernos optimizan las expresiones.
Optimización (15 min): La eficiencia viene del diseño, no de los trucos. Programación dinámica vs. Greedy. Hablar de la idea de "evitar trabajo" y de la importancia de los profilers para saber dónde está el problema real.

Actividad 21: Diseño de plan de ataque

En grupos de proyecto, apliquen los 3 Pilares a su proyecto.

1. Plan de Verificación

Define 1 Caso de Prueba Crítico (Peor Caso):

  • Entrada: (Ej. 0, el valor máximo, un texto malformado)
  • Resultado Esperado: (Ej. Mensaje de error, no falla, devuelve `null`)

2. Plan de Depuración

Identifica 1 Problema a Manejar:

  • Problema: (Ej. Lectura de archivo inexistente, intento de división por cero, objeto `null` inesperado).
  • Acción: (Ej. Implementar `try-catch`, mostrar mensaje de error amigable al usuario).

3. Plan de Optimización

Identifica 1 Mejora de Algoritmo o Código:

  • Sección: (Ej. El bucle de búsqueda de datos, la gestión de memoria de una estructura).
  • Acción: (Ej. Reemplazar una lista por un *hash map*, refactorizar una función con más de 20 líneas).
Plan de Ataque (10 min): Este es el momento de aplicación. Los estudiantes deben anotar sus 3 puntos débiles y cómo los atacarán. Esto asegura que la lección se aplique inmediatamente a su trabajo.
